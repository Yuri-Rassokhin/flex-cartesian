Unreliable data.

Real systems are defined by multiple independent parameters. Each configuration is just one observation in a much larger space. However, exploration is often performed through isolated runs: ad-hoc benchmarks, biased experiments, manual comparisons, fragmented logs, and untracked data.

This leads to fragmented, unreliable, and incomplete knowledge. The system is never understood as a whole — only as a set of disconnected observations, stitched together through assumptions rather than insight.

What if we stopped running experiments… and started exploring the system itself?

FlexCartesian treats a system not as a set of scattered measurements, but as a holistic function that behaves over space with dimenions defined by system parameters. This shift in perspective from blind probes to parameteric analysis allows us to move from fragmented data points to a structured understanding of how the system behaves across different configurations. It enables us to identify patterns, trade-offs, and bottlenecks that would be invisible in isolated runs.

Ruby DSL

Readable enough for humans, precise enough for systems.

FlexCartesian gives you a compact way to describe dimensions, functions, and execution logic. It is built for scenarios where you want engineering clarity instead of opaque optimisation pipelines.

  • Dimensions and values
  • Conditions and filters
  • Functions bound to live evaluation logic
  • Structured outputs for downstream analysis
fc = FlexCartesian.new

fc.dimension :batch_size, [1, 2, 4, 8]
fc.dimension :gpu_count,  [1, 2, 4]
fc.dimension :precision,  [:fp16, :bf16]

fc.function :latency_ms do |params|
  benchmark_model(params).latency_ms
end

fc.function :cost_per_token do |params|
  infra_cost(params) / benchmark_model(params).tokens
end

fc.run
fc.export "results.csv"

Born in field work

FlexCartesian emerged from real-world AI infrastructure work, where system behaviour depends on multiple interacting parameters (hardware, storage, network, model configuration). Traditional tools failed to provide a unified abstraction to explore such systems.

Applied in critical systems

  • Performance modelling across distributed systems
  • AI architecture benchmarking: GPU, storage, and network trade-offs
  • Architecture decision support for enterprise AI deployments

What this enables

  • Discovery of optimal configurations without exhaustive testing
  • Identification of system bottlenecks across dimensions
  • Bridging theoretical modelling with real measurements
